# DATABASE.md: Bitemporal Property Graph & Metamodel Specification

## 1. Executive Summary & Design Rationale
This system is a high-integrity Digital Twin engine. It is designed to model complex physical and logical networks (e.g., electrical grids) where historical accuracy and topological connectivity are mission-critical.

### Core Design Decisions
* **Bitemporality:** We use `valid_from` and `valid_to` to allow "Point-in-Time" reconstruction. 
* **Property Graph via RDBMS:** We use PostgreSQL to gain the ACID compliance of a relational database while using JSONB and Object and Relationship tables to provide the flexibility of a Graph database.
* **Metadata-Driven Architecture:** The database is "self-describing." The Data Dictionary defines the rules for objects, relationships, and  the SQL views used to expose data for spatial visualisations.
* **Decoupled State:** Operational volatility (e.g., a switch opening) is stored in a `state` table. Asset data (e.g., the switch's serial number) is stored in the `object` table. This prevents "Version Explosion" in the master data.

---

## 2. The Data Dictionary (The Metamodel)
The dictionary describes the objects and relationships and how inheritance works, it provids a definition of attributes within objects and and defins views from the core model.

### 2.1 Class Inheritance & Overrides
Classes (Object types) are recursive. For axample an `11kV_Breaker` inherits attributes from `Breaker`, which inherits from `Switch`.
* **Resolution Logic:** When determining required attributes, the system must traverse the class tree. A configuration at a lower level (child) overrides the same attribute at a higher level (parent).
* **Term Types:** The dictionary governs three domains: `OBJECT` (the entity), `RELATIONSHIP` (the link), and `STATE` (the status profile).

### 2.2 View & Presentation Layer
To support spatial visulisations (vi PostGIS), we define "View Specs." These are not static SQL; they are instructions to flatten JSONB attributes into standard columns.
* **Display Names:** Both views and columns have `display_name` fields to ensure UI legends are human-readable, decoupling them from SQL-safe aliases.
* **Batch Refresh:** Materialized views are refreshed via external orchestration, not triggers, to maintain performance during bulk master data updates.

## 3. Implementation Instructions for Claude (Sonnet)

### 3.1 Temporal Integrity
Every query generated by the model **must** include the temporal filter. If no specific time is provided, the model must default to the "Active" state.
* **Filter Logic:** `valid_from <= :T AND (valid_to > :T OR valid_to IS NULL)`

### 3.2 Ingestion & Change Detection
Updates to the `object` table follow a "Terminate and Insert" pattern.
1.  Calculate the `hash` (MD5) of the incoming `attributes` JSONB.
2.  Compare with the current active record.
3.  If a change exists: update the old record's `valid_to` to the current timestamp and insert the new version.

### 3.3 Topology & Traversal
* **Connectivity:** Relationships of type `edge` represent functional flow.
* **Composition:** Relationships of type `composition` represent the physical container hierarchy.

---

## 4. Background & Contextual Information
* **Namespace/Identity:** This is the immutable natural key. While a `uuid` might change per version, the `(namespace, identity)` pair identifies the asset for its entire life.
* **Spatial Data:** We utilize PostGIS `GEOMETRY` types. `geo_geometry` is for real-world coordinates; `sch_geometry` is for logical schematic layouts. `sch_geometry` is currently reserved for future use and visualisatiosn should use `geo_geometry` initially.

## 5. Consolidated Code & DDL Reference

### Listing 5.1: Full Relational Schema
```sql
CREATE SCHEMA IF NOT EXISTS network_model;
SET search_path TO network_model;

-- I. DATA DICTIONARY
CREATE TABLE class_definition (
    class_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    parent_class_uuid UUID REFERENCES class_definition(class_uuid),
    namespace TEXT NOT NULL,
    name TEXT NOT NULL, 
    term_type TEXT CHECK (term_type IN ('OBJECT', 'RELATIONSHIP', 'STATE')),
    is_abstract BOOLEAN DEFAULT FALSE,
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ,
    CONSTRAINT unique_active_class UNIQUE (namespace, name, term_type, valid_to)
);

CREATE TABLE attribute_definition (
    attr_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    namespace TEXT NOT NULL,
    name TEXT NOT NULL, 
    data_type TEXT NOT NULL,
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ,
    CONSTRAINT unique_active_attr UNIQUE (namespace, name, valid_to)
);

CREATE TABLE class_attribute_config (
    config_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    class_uuid UUID NOT NULL REFERENCES class_definition(class_uuid),
    attr_uuid UUID NOT NULL REFERENCES attribute_definition(attr_uuid),
    is_required BOOLEAN DEFAULT FALSE,
    default_value TEXT,
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ
);

-- II. VIEW SPECIFICATIONS
CREATE TABLE view_definition (
    view_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    namespace TEXT NOT NULL,
    view_name TEXT NOT NULL,         
    display_name TEXT NOT NULL,      
    is_materialized BOOLEAN DEFAULT FALSE,
    refresh_group TEXT,              
    class_uuid UUID REFERENCES class_definition(class_uuid),
    discriminator_filter TEXT,
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ,
    CONSTRAINT unique_active_view UNIQUE (namespace, view_name, valid_to)
);

CREATE TABLE view_column_spec (
    column_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    view_uuid UUID NOT NULL REFERENCES view_definition(view_uuid),
    source_path TEXT NOT NULL,       
    alias TEXT NOT NULL,             
    display_name TEXT NOT NULL,      
    cast_type TEXT,                  
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ
);

-- III. INSTANCE DATA
CREATE TABLE object (
    uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    namespace TEXT NOT NULL,
    identity TEXT NOT NULL,          
    class_uuid UUID NOT NULL REFERENCES class_definition(class_uuid),
    discriminator TEXT,              
    geo_geometry GEOMETRY,
    sch_geometry GEOMETRY,
    attributes JSONB DEFAULT '{}',
    hash TEXT,                       
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ,
    CONSTRAINT unique_active_object UNIQUE (namespace, identity, valid_to)
);

CREATE TABLE relationship (
    rel_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_uuid UUID NOT NULL REFERENCES object(uuid),
    target_uuid UUID NOT NULL REFERENCES object(uuid),
    class_uuid UUID NOT NULL REFERENCES class_definition(class_uuid),
    rel_type TEXT CHECK (rel_type IN ('edge', 'composition')),
    attributes JSONB DEFAULT '{}',
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ
);

CREATE TABLE state (
    state_uuid UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    object_uuid UUID NOT NULL REFERENCES object(uuid),
    class_uuid UUID NOT NULL REFERENCES class_definition(class_uuid),
    namespace TEXT NOT NULL,
    identity TEXT NOT NULL,
    state_data JSONB NOT NULL,
    valid_from TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMPTZ
);
```

### Listing 5.2: Example Recursive Inheritance Query
```sql
WITH RECURSIVE lineage AS (
    SELECT class_uuid, parent_class_uuid, 1 as level
    FROM class_definition WHERE class_uuid = :target_uuid
    UNION ALL
    SELECT c.class_uuid, c.parent_class_uuid, l.level + 1
    FROM class_definition c
    JOIN lineage l ON c.class_uuid = l.parent_class_uuid
)
SELECT a.name, config.is_required
FROM class_attribute_config config
JOIN attribute_definition a ON config.attr_uuid = a.attr_uuid
WHERE config.class_uuid IN (SELECT class_uuid FROM lineage)
ORDER BY level ASC; 
```
